## 1. 專案目標與範圍

### 1.1 專案目標
建立一個自動化管理系統，協助分校主任：

- 精準推演學生學習進度
- 管理學生剩餘學習時數
- 確認每日 / 逾期待確認進度
- 管理學習矩陣中的庫存 / 訂購狀態
- 自動彙整跨分校書籍訂購需求
- 降低人工計算與行政整理成本

### 1.2 使用角色
主要使用者為：

- 分校主任
- 行政管理人員

### 1.3 系統管理範圍
系統需支援：

- 3 個分校：`延壽 / 安和 / 大直`
- 約 300 名學生
- 桌面版與手機版操作
- Google Sheets 作為主要資料來源 / 儲存目標

## 2. 核心領域模型與名詞定義

### 2.1 MPM 書號格式
MPM 書號格式為：

`{Level}{Grade}{No}`

其中：

- `Level`：難度層級
  - `GK`
  - `GV`
  - `GA`
- `Grade`：年級，`0 ~ 6`
- `No`：書號流水號，但不同 Level 的合法編號區間不同

### 2.2 合法書號規則

#### GK
- `01 ~ 40`
- `41 ~ 80`
- `81 ~ 88`
- `89 ~ 96`

#### GV
- `01 ~ 24`
- `33 ~ 56`
- `65 ~ 88`
- `89 ~ 96`

#### GA
- `01 ~ 24`
- `33 ~ 56`
- `65 ~ 80`

補充實作約束：

- 合法書號的單一權威來源為前端 `RANGES` 常數（`react-app/src/constants.js`）。
- 書號合法性驗證（含 `validateMPMBook`）與 UI 輸入上限必須共用同一份 `RANGES`，禁止另行硬編碼上限。

### 2.3 套書規則
向 MPM 廠商訂購書籍時，以 **8 本為 1 套**。

例如：

- `GK201 ~ GK208`
- `GV433 ~ GV440`
- `GA265 ~ GA272`

套書範圍規則為：

- `1 ~ 8`
- `9 ~ 16`
- `17 ~ 24`
- 以此類推

### 2.4 學習進度表狀態
學習進度表 `status` 定義如下：

- `pending`
  - 已推演或已儲存，但尚未確認
- `match`
  - 符合進度
- `behind`
  - 落後進度
- `ahead`
  - 超前進度
- `none`
  - 無有效進度狀態，通常僅作過渡或空值處理

補充正規化規則：

- `learned` 視為 `match`
- `planned / pending / planed` 視為 `pending`

### 2.5 書籍訂購狀態
`book_order_state` 僅管理以下狀態：

- `needOrder`
- `inStock`
- 空值 `''`

不屬於 `book_order_state` 的狀態：

- `planned / pending`
- `learned / match / behind / ahead`
- `none`

上述狀態屬於學習進度表語意，不得寫入 `book_order_state`。

### 2.6 學期日期模型
系統設定中的學期日期必須使用 **完整日期**，不可只存月日。

支援輸入格式：

- `YYYY/MM/DD`
- `YYYY-MM-DD`

系統內部統一正規化為：

- `YYYY-MM-DD`

學期設定欄位為：

- 上學期開始日期 `upTermStart`
- 上學期結束日期 `upTermEnd`
- 下學期開始日期 `downTermStart`
- 下學期結束日期 `downTermEnd`

## 3. 核心業務邏輯

### 3.1 學習進度推演邏輯

#### 3.1.1 計算單位
學習進度以「本」為單位，支援半本進度：

- `1/2`
- `2/2`

例如：

- `GK378(1/2)`
- `GK378(2/2)`

#### 3.1.2 輸入參數
推演至少需使用以下參數：

- 起始書號
- 每週進度表（Schedule）
- 學習速度（Speed）
- 目前剩餘學習時數
- 起始日期
- 結束日期（可空）

#### 3.1.3 終止條件
推演在以下任一條件成立時停止：

1. 到達該 Level / Grade 的最後一本合法書號
2. 到達設定的結束日期

#### 3.1.4 時數耗盡規則
當累計消耗時數超過目前剩餘學習時數時：

- 推演不停止
- 需明確標示時數已用盡分界
- 最終剩餘學習時數可為負值

此設計用於：

- 提前預警續費需求
- 模擬未補時數下的未來進度

### 3.2 學習進度確認規則
當學習進度表存在 `pending` 列時，主任可進行確認。

確認目標列選取規則：

- 僅從 `date <= 今天` 的 `pending` 列中選取
- 若有多筆，取「最接近今天」的那一筆（日期最大）
- `date > 今天` 的 `pending` 不進入「今日進度確認」

確認時可調整：

- 實際時數
- 實際書號
- 學習狀態
- 備註

限制：

- 必須選擇狀態：`match / behind / ahead`
- 至少填入一個實際書號
- 時數需為合法數值
- 若狀態為 `behind`，備註必填

確認成功後：

- 該列由 `pending` 更新為 `match / behind / ahead`
- 寫入 `confirmedAt`
- 扣減學生剩餘學習時數
- 同步更新學生設定與學習進度表
- 若本次確認書號中存在非 `inStock`，需先警告使用者；使用者同意後，先將該批書號強制改為 `inStock`，成功後才可完成確認

`confirmedAt` 寫入規範：

- 寫入格式為：`yyyy/M/d 上午|下午 h:mm:ss`
- `confirmedAt` 欄位以文字格式寫入，避免被 Spreadsheet 自動轉為其他日期顯示格式

### 3.2.1 教材查詢口徑（某學生、某日期）
- 資料來源：學習進度表（`scheduleTable`）
- `match / behind / ahead`：已確認（實際發生）
- `pending`：未確認（預計）
- 預設查詢口徑應以「已確認」為主；若需預排資訊，可切換顯示 `pending`

### 3.3 學期範圍邏輯
所有與學期相關的顯示與運算，皆以完整日期區間判斷，不再使用 `MM/DD` 推估跨年學期。

學習矩陣範圍選項固定為：

- `上學期`
- `下學期`
- `上＋下學期`

不再使用：

- `全部`
- `本學期`

定義如下：

- `上學期`：日期位於 `upTermStart ~ upTermEnd`
- `下學期`：日期位於 `downTermStart ~ downTermEnd`
- `上＋下學期`：上述兩段區間聯集，代表整個學年

### 3.4 學習矩陣預設範圍規則
矩陣預設範圍需依今日日期決定：

- 今日位於上學期區間 → 預設 `上學期`
- 今日位於下學期區間 → 預設 `下學期`
- 今日不在上述區間，或學期設定不足 → 預設 `上＋下學期`

若使用者在本次 session 中已手動切換過範圍：

- 優先保留手動選擇
- 不在同一次 session 中被自動覆蓋

### 3.5 自動訂購與 K 規則

#### 3.5.1 K 值來源
K 值採「個人優先，全域後備」：

- 個人：`orderAlertGapKByPerson`
- 全域：`setting.orderAlertGapK`

規則：

- 若 `orderAlertGapKByPerson > 0`，優先使用個人值
- 否則使用全域值

#### 3.5.2 K 判斷單位
K 判斷以 **唯一主書號** 為單位，不以 half token 數量計算。

例如：

- `GK378` → 算 1 本
- `GK378(1/2)` → 算 1 本
- `GK378(2/2)` → 算 1 本
- `GK378(1/2) + GK378(2/2)` → 仍算同 1 本

#### 3.5.3 K window 起點
K window 起點基準 `maxLearnedNo` 依優先序決定：

1. 優先使用目前使用書號
2. 若無法解析，fallback 為最後已學過書號
3. 若都不存在，視為 `0`

#### 3.5.4 K window 掃描規則
由 `maxLearnedNo + 1` 開始往後掃描合法書號，只納入：

- 合法書號
- 尚未 learned
- 屬於 `planned / pending`

取 **最多 K 個唯一主書號** 作為 K window。

#### 3.5.5 K window 判斷
- 若 K window 為空 → 不提示
- 若 `0 < window < K` → 以實際找到的 pending 判斷是否都為 `inStock`
- 若 `window = K` → 一般情況照常判斷

#### 3.5.6 套書提示規則
當 K window 不為空時：

- 需將其中書號映射到套書
- 若某套中任一格不是 `inStock`
  - 顯示「須要訂購此套」
- 若 K window 內所有格都為 `inStock`
  - 不顯示提示

#### 3.5.7 reconcile 自動清理規則
若某套目前已不需購買：

- 該套內原本的 `needOrder` 應清為空值 `''`

但以下狀態不得被自動覆寫：

- `inStock`

## 4. 前端 UI / UX 規格

### 4.1 整體架構
系統採三層式資訊架構：

1. 分校與模組控制區
2. 學生列表區
3. 學生詳細資訊與互動區

並區分：

- 桌面版
- 手機版

### 4.2 頂部導覽列

#### 桌面版
TopNav 固定提供：

- 分校切換：`延壽 / 安和 / 大直`
- 模組切換：`📚 學習進度`、`📦 書籍訂購`
- 系統按鈕：`⚙️ 系統設定`、`🔄 重新整理`

#### 手機版
手機版 TopNav 僅保留：

- `☰ 選單`

所有主要操作集中於 Drawer。

### 4.3 手機版 Drawer 規格
Drawer 內容包含：

- 分校下拉選單
- 模組切換
- 系統設定
- 重新整理
- 搜尋 / 篩選
- 學生清單

Drawer 規則：

- 必須支援 `overflow-y: auto`
- 必須限制 `overflow-x: hidden`
- scroll bar 僅存在於 Drawer 內部
- 關閉 Drawer 後，主畫面應聚焦在目前學生主要資訊

### 4.4 學生列表規格

#### 桌面版
左側顯示學生清單，支援：

- 搜尋學生姓名
- 搜尋年級文字
- 依 Level 分組
- 新增學生
- 狀態篩選

分組與排序規則：

- 分組順序固定為：`GK → GV → GA`
- 同群內固定以「學生姓名遞增」排序（中文語系排序）
- 排序僅影響顯示順序，不改變篩選條件與分組邏輯

#### 手機版
學生清單收納在 Drawer，需採手機友善扁平列表，不強調大卡片設計。

點選學生後：

- Drawer 自動關閉

### 4.5 學生列表 badge 與 filter 規則
badge 僅有兩種：

- `🕒 今待確認`
- `⏰ 逾期`

計算方式：

- 以 `scheduleTable` 中 `status = pending` 的列即時計算
- 僅看 `date <= 今天`
- 取最早一筆待確認日期：
  - `< 今天` → `⏰ 逾期`
  - `= 今天` → `🕒 今待確認`

優先順序：

- 若同時符合，僅顯示 `⏰ 逾期`

filter 保留：

- `全部`
- `⚠ 時數預警`
- `🕒 今待確認`
- `⏰ 逾期未確認`

不再提供：

- `缺書`

### 4.6 學生詳細區規格
學生詳細區需顯示：

- 最新已確認書號
- 今日待確認書號（預計）
- 剩餘學習時數
- 已購時數
- 總學習時數
- 預計耗盡日
- 時數狀況
- 學習矩陣-訂購預警窗口 K
- 需要套書

「最新已確認書號」來源規則：

1. 優先取學習進度表中 **最新日期且已確認**（`match / behind / ahead`）那一列的最後一本書號。
2. 若無任何已確認書號，才 fallback 使用學生主檔 `level + grade + confirmedNo`。
3. 若以上兩者都無法形成合法書號，顯示 `—`。
4. 側欄學生列表與學生詳細區必須使用同一套計算邏輯，避免顯示不一致。

「今日待確認書號（預計）」來源規則：

1. 取 `date <= 今天` 且 `status = pending` 的列中，最接近今天（日期最大）那一列。
2. 顯示該列 `books`（多本以 `、` 串接）。
3. 若無符合列，顯示 `無`。

`需要套書` 顯示規則：

- 顯示為套書區間，例如 `GK201~GK208`
- 多套以 `、` 串接
- 無資料時顯示 `無`
- 若 schedule 尚未載入完成，顯示 `載入進度表中…`

詳細區另需提供：

- `編輯學生`
- `生成學習進度表`
- `儲值時數`

### 4.7 儲值時數規格
儲值時數：

- 僅接受大於 `0` 的整數
- 可輸入備註
- 成功後即時更新：
  - 剩餘學習時數
  - 時數狀況

### 4.8 今日進度確認區
桌面版：

- 可維持一般區塊式配置

手機版：

- 改為 stacked arrangement
- 上下堆疊顯示：
  - 日期
  - 實際時數
  - 實際書號
  - 學習狀態
  - 備註
  - 確認按鈕

### 4.9 學習進度表（全覽）
桌面版：

- 維持表格式顯示

手機版：

- 改為 Expandable Row / 卡片式展開

手機版摘要欄位先顯示：

- 日期
- 星期
- 時數
- 學習狀態

展開後顯示：

- 書號
- 備註
- 確認時間

目的：

- 避免手機畫面橫向超寬

### 4.10 書號學習矩陣
學習矩陣提供：

- Level 切換：`GK / GV / GA`
- 範圍切換：`上學期 / 下學期 / 上＋下學期`
- `🔄 同步學習矩陣狀態`

視覺符號：

- `綠底𒊹`：整本已完成
- `綠底 ◐`：半本已完成
- `白底 ○`：已排入學習進度但尚未確認
- `黃底 ○`：有庫存
- `紅底 ○`：需訂購 / 無庫存

矩陣中的：

- 已學過書號
- 已排定書號
- 訂購提示

皆需使用相同 scope 過濾後的資料計算，禁止出現 scope 不一致。

畫面需明確顯示：

- 目前套用範圍文字
- 對應日期區間

### 4.11 矩陣互動規則

#### 點單格
僅切換：

- `inStock <-> ''`

不得直接由單格切成 `needOrder`。

#### 點「訂購此套」
- 針對該套合法書號
- 將空白 / 未標記改為 `needOrder`
- 若原格為 `inStock`，不得覆寫

#### 點「同步學習矩陣狀態」
- 將目前學生矩陣中的 `inStock / needOrder / ''` 同步到後端 `book_order_state`

### 4.12 書籍訂購模組
`📦 書籍訂購` 模組為查詢頁，不直接修改狀態。

顯示欄位：

- Level
- 年級
- 套書範圍
- 需訂學生數
- 學生名單

篩選條件：

- 搜尋：學生姓名或 Level
- 時間：`全部 / 30天 / 90天 / 半年 / 自選`
- 自選日期區間
- 清除篩選

若無資料：

- 顯示空狀態提示

### 4.13 新增 / 編輯學生
Modal 欄位包含：

- 學生編號
- 學生姓名
- 分校
- 國小
- Level
- 年級
- 起始書號
- 速度
- 剩餘學習時數
- 已購時數
- 訂購預警 K
- 每週時程（週一～週日）

驗證規則：

- 學生編號必填
- 學生姓名必填
- 分校必選
- 起始書號需大於 `0`
- 剩餘學習時數不可為負數
- 已購時數不可為負數

編輯模式：

- 學生編號欄位鎖定
- 可刪除學生

### 4.14 系統設定 Modal
設定欄位包含：

- 上學期開始日期
- 上學期結束日期
- 下學期開始日期
- 下學期結束日期
- 每頁筆數
- 時數不足預警
- 載入今天前幾天進度
- 載入今天後幾天進度
- 學習矩陣-預警無庫存書的提示窗口大小 K
- 學習進度表-提前提醒需要新套書的天數

規則：

- 日期欄位支援 `YYYY/MM/DD`、`YYYY-MM-DD`
- 儲存時自動正規化為 `YYYY-MM-DD`
- 儲存成功後前端立即採用新設定
- 儲存中不可直接關閉 Modal

## 5. 前端載入、快取與同步規則

### 5.1 首屏載入
App 啟動時：

1. 顯示整體 loading 狀態
2. 載入預設分校 `延壽` 學生
3. 載入 global settings
4. 載入 `book_order_state`

完成後：

- 左側只顯示目前 branch 學生
- 預設選取該 branch 第一位學生
- 再按需載入該學生學習進度表

### 5.2 branch lazy load
切換 branch 時：

1. 先檢查前端是否已有該 branch 學生資料
2. 若尚未載入：
   - 顯示 loading
   - 向 Google Sheets 抓取該 branch 學生
3. 若已載入：
   - 直接切換顯示
   - 不重新抓學生 API
4. 若目前選取學生不屬於該 branch：
   - 自動改選該 branch 第一位學生
   - 若無學生則 `selectedStudent = null`

### 5.3 學習進度表部分載入
學習進度表採日期窗口部分載入。

預設窗口由 settings 控制：

- `scheduleLoadPastDays`
- `scheduleLoadFutureDays`

前端讀取單一學生進度時，需帶入：

- `startDate = today - scheduleLoadPastDays`
- `endDate = today + scheduleLoadFutureDays`

### 5.4 branch 內 schedule preload
當使用者在某 branch 第一次選到任一學生時：

- 以前端目前日期窗口
- 一次 preload 該 branch 全部學生 schedule

若該 branch schedule 正在載入：

- 詳細區顯示 `scheduleLoading`
- 不可誤顯示成「沒有資料」

若已載入相同日期窗口：

- 直接使用前端快取

若日期窗口改變：

- 重新載入目前 branch 各學生 schedule

### 5.5 設定變更與快取失效
使用者修改 settings 並儲存後：

1. settings 先保存到 Google Sheets
2. 前端更新目前生效 settings
3. schedule 快取視為失效
4. 重新依新日期窗口載入 schedule

### 5.6 rowId 主鍵規則
由於同一學生、同一天可能有多筆學習進度資料，因此每一列需有唯一 `rowId`。

規則：

- 每筆學習進度列都有唯一 `rowId`
- 舊資料若沒有 `rowId`，系統自動補 UUID
- 新增推演列時，前端先產生 `rowId`
- 寫入 Google Sheets 時一併保存
- 確認進度時，優先用 `rowId` 精準更新

### 5.7 optimistic UI 與同步鎖定
矩陣相關操作採 optimistic UI：

- 先更新本地 UI
- 不在每次點格時立即送 Sheet
- 改由「同步學習矩陣狀態」做整位學生全量同步

同步期間以下元件需 disabled：

- 同步學習矩陣狀態按鈕
- 矩陣格子
- 訂購此套按鈕

成功或失敗後都必須解鎖。

### 5.8 還原流程操作鎖定
在學生詳情區執行「還原成未確認」期間，需啟用全域操作鎖定：

- 禁用其他「還原」按鈕（桌面 / 手機）
- 禁用「今日進度確認」按鈕
- 禁用「調整時數」按鈕
- 禁用「更新」與分頁按鈕

還原成功或失敗後皆需解鎖。

操作欄按鈕規則：

- `pending` 列：顯示「刪除」
- `match / behind / ahead` 列：顯示「還原」
- 同一列不會同時顯示「刪除」與「還原」

## 6. 儲存與後端寫入規則

### 6.1 套用設定
套用設定時，同步目標：

- `學生設定`

同步欄位包含：

- `level`
- `grade`
- `confirmedNo`
- `speed`
- `currentRemainingHours`
- `confirmedHours`
- `orderAlertGapKByPerson`
- `mon ~ sun`

不同步：

- 起始日期
- 結束日期

因為兩者屬於推演條件，而非學生主檔設定。

### 6.2 執行推演
執行推演時：

- 只讀取目前畫面上最新的有效設定
- 只顯示結果
- 不直接寫入學習進度表

### 6.3 儲存為新的學習進度表
按下「儲存為新的學習進度表」時：

#### 保留規則
- 採區間覆寫（`replaceStartDate = 今天`、`replaceEndDate = 空`）
- 僅刪除覆寫區間內未確認列（`pending / none`）
- 覆寫區間內已確認列（`match / behind / ahead`）保留
- `日期 < 今天` 的列不在覆寫區間內，天然保留

#### 新增規則
- 本次推演結果全部重新插入
- 新插入列狀態一律為 `pending`
- 即使同一天已有已確認列，仍可插入新的 `pending`
- 若同一次寫入中，針對同一 `studentId` 需新增多筆列（append），新列必須先依 `date` 升冪後再落地
- 同日多筆時，維持原輸入順序（穩定排序）

#### 排序規則
1. 先依日期排序
2. 同日期時 `pending` 排前
3. 已確認列排後

### 6.4 saveSchedule 區間覆寫規則
「儲存學習進度推演（save and update）」採 **整個未來覆寫**（同一 `studentId`）。

`saveSchedule` 可帶入：

- `replaceStartDate`
- `replaceEndDate`

規則：

1. `replaceStartDate` 固定為今天（`today`）
2. `replaceEndDate` 可留空，代表未來不設上限（直到最後一筆）
3. 今日與未來中：
   - `match / behind / ahead`（已確認）保留
   - `pending / none`（未確認）刪除
4. 再寫入本次新推演結果（狀態一律 `pending`）
5. 目標是避免舊 `pending` 殘留，造成看起來只新增不刪除

### 6.5 確認進度寫回規則
確認進度時同步目標：

- `學習進度表`
- `學生設定`

流程：

1. 更新對應列狀態
2. 寫入 `confirmedAt`
3. 更新學生 `confirmedNo`
4. 更新學生 `currentRemainingHours / confirmedHours`

### 6.5.1 還原成未確認（undoConfirmProgress）
為避免誤填時數或狀態，系統提供「還原成未確認」功能。

規則：

1. 僅允許 `match / behind / ahead` 列還原成 `pending`
2. 不限制 7 天，凡符合狀態即可還原
3. 還原時：
   - `status -> pending`
   - `confirmedAt -> 空`
   - `hours / books / note` 保留原值（不清空）
   - 該列 `hours` 加回學生 `currentRemainingHours / confirmedHours`
4. 還原不回滾 `book_order_state` 的 `inStock`（庫存狀態視為獨立管理）
5. 還原操作需寫入時數異動紀錄（type: `undo_confirm`）
6. 還原成功後，「今日進度確認」表單需自動帶回該列 `書號 / 時數 / 備註`，供主任快速修改後再確認

### 6.5.2 刪除待確認列（deletePendingProgressRow）
為處理誤植的待確認資料，系統提供刪除 `pending` 列功能。

規則：

1. 僅允許刪除 `pending` 列
2. 刪除時不扣時數，也不回補時數
3. 刪除前需二次確認
4. 刪除後該列不再進入「今日進度確認」

### 6.6 saveBookOrderStates 規則
對每筆 `{ studentId, bookCode, state }`：

1. `bookCode` 正規化為大寫
2. 若 `state` 非 `needOrder / inStock`，視為空值 `''`
3. 以 `(studentId, bookCode)` 查找既有列
   - 找到：更新 `state / updatedAt / updatedBy`
   - 找不到且狀態為 `needOrder / inStock`：新增新列
   - 找不到且狀態為 `''`：不新增空白列

同步語意採 **upsert**。

### 6.7 同步矩陣資料來源
同步學習矩陣狀態時：

- 直接掃描 `bookOrderStateMap` 中該學生所有 key
- `needOrder / inStock` 一律送出
- 曾存在於 Sheet 但本地已清空者，送出 `state = ''` 清理
- 若某 `needOrder` 已不在目前有效 `bookAlertSetCodes`（脫離當前 K window 需求），也必須送出 `state = ''` 做**實際清理**

## 8. 驗收條件

### 8.1 學習進度表驗收
1. 推演結果可正確依時程、速度、剩餘時數推算
2. 終止條件符合最後一本到達或結束日期到達
3. 剩餘時數可為負
4. 儲存時保留舊的已確認資料
5. 新推演列狀態一律為 `pending`
6. 同一天可存在已確認列與新 `pending` 列
7. 確認進度時優先用 `rowId` 精準更新
8. 起始書號輸入上限需依 Level 動態變化（由合法書號範圍推導，不可固定常數）

### 8.2 branch / schedule 載入驗收
1. 首屏只載入單一 branch 學生
2. branch 資料採 load once
3. 學習進度表採部分日期區間載入
4. branch 首次點學生後會 preload 該 branch 全部學生 schedule
5. 日期窗口變更時會失效快取並重新載入

### 8.2.1 學生清單排序驗收
1. 分組順序固定為 `GK → GV → GA`
2. 同群內學生依姓名遞增顯示
3. 在搜尋/篩選條件改變時，排序結果仍維持姓名遞增

### 8.3 矩陣與訂購驗收
1. 點單格只切換 `inStock <-> ''`
2. 點「訂購此套」後，空白格可變 `needOrder`
3. `inStock` 不會被自動訂購或 reconcile 覆寫
4. 同步後 `book_order_state` 與 UI 狀態一致
5. 沒有列但狀態為 `needOrder / inStock` 時，必須新增列
6. UI 清空且 Sheet 已有該書號時，需更新為空值 `''`
7. 若某套現在不用買，舊 `needOrder` 會被清空
8. 同步期間按鈕與格子會 disabled，返回後恢復
9. 過期 `needOrder` 必須實際清到 Sheet（非僅 UI 隱藏）
10. 例：僅排 `GK101~GK103` 並完成確認後，不得再殘留顯示 `GK101~GK108` 待訂

### 8.4 K 規則驗收
1. K 判斷採個人優先、全域後備
2. K 以唯一主書號計算，half 不重複算本數
3. K window 起點優先使用目前使用書號
4. `window = 0` 不提示
5. `0 < window < K` 以實際找到的 pending 判斷
6. `window = K` 依一般規則判斷
7. 近期需訂購套書可正確映射到套書區間

### 8.5 手機版 UI 驗收
1. 手機版功能選單集中於 Drawer
2. Drawer 可獨立垂直捲動
3. 點選學生後 Drawer 自動關閉
4. 手機版學習進度表改為 Expandable Row / 卡片式展開
5. 手機版今日進度確認改為上下堆疊
6. 主畫面聚焦目前學生資訊，不重複佔用控制區

### 8.6 學期與矩陣範圍驗收
1. 學期日期支援 `YYYY/MM/DD`、`YYYY-MM-DD`
2. 儲存時統一正規化為 `YYYY-MM-DD`
3. 矩陣範圍固定為：
   - 上學期
   - 下學期
   - 上＋下學期
4. 已學過 / 已排定 / 訂購提示使用相同 scope 計算
5. 預設 scope 會依今日日期自動選擇
6. 同一 session 內手動切換過 scope 後，不被自動覆蓋

## 11. 一句話版規則摘要

- **套用設定**：更新學生主檔設定並同步 Google Sheets
- **執行推演**：只計算，不落地
- **儲存為新的學習進度表**：保留已確認資料，只覆寫指定區間內未確認列
- **確認進度**：將 `pending` 正式轉為 `match / behind / ahead`
- **rowId**：學習進度表每列唯一鍵
- **首屏載入**：只先載入單一 branch
- **schedule 載入**：首次點 branch 學生後 preload 同 branch 全部學生
- **待確認提示**：由前端即時計算 badge 與 filter
- **需要套書**：顯示於學生詳細區，以套書區間格式呈現
- **矩陣同步**：採 optimistic UI，並以整位學生全量同步 `book_order_state`
