## 📋 需求背景

### 原始問題
修改學生基本資料（例如備註欄）後，系統會重新載入整個學生清單，造成：
- 不必要的 API 呼叫
- 額外的資料傳輸
- 較差的使用者體驗

### 使用者疑問
> 為什麼修改學生基本資料，如備註欄，就要重讀學生清單？

---

## 🎯 解決方案

### 方案 A：後端回傳完整學生資料 + 前端直接更新（已採用）

**優點**：
- 效能最佳，避免重新載入整個學生清單
- 實作簡單，後端只需多回傳一個 student 物件
- 可靠性高，直接使用後端回傳的最新資料

**實作邏輯**：
1. 後端 `updateStudent()` 成功更新後，回傳完整的學生資料
2. 前端收到後，直接更新 `students` 狀態中的該學生
3. 不觸發 `setRefreshKey`，避免重新載入

```javascript
// 修改前
async function handleSaveStudent(payload, mode) {
  try {
    setStudentManageSaving(true)

    if (mode === 'edit') {
      await updateStudent(payload)  // ← 不保存回傳結果
    } else {
      await addStudentApi(payload)
    }

    setCurrentBranch(payload.branch || currentBranch)
    setSelectedId(payload.id)
    setStudentManageOpen(false)
    setStudentManageTarget(null)
    setRefreshKey((prev) => prev + 1)  // ← 觸發重新載入整個清單
    return { ok: true }
  } catch (error) {
    return { ok: false, message: error.message || '學生資料儲存失敗' }
  } finally {
    setStudentManageSaving(false)
  }
}

// 修改後
async function handleSaveStudent(payload, mode) {
  try {
    setStudentManageSaving(true)

    let result  // ← 保存 API 回傳結果
    if (mode === 'edit') {
      result = await updateStudent(payload)
    } else {
      result = await addStudentApi(payload)
    }

    setCurrentBranch(payload.branch || currentBranch)
    setSelectedId(payload.id)
    setStudentManageOpen(false)
    setStudentManageTarget(null)

    // 方案 A：使用後端回傳的學生資料直接更新，避免重新載入整個列表
    if (mode === 'edit' && result?.student) {
      setStudents((prev) =>
        prev.map((s) => (s.id === result.student.id ? { ...s, ...result.student } : s))
      )
    } else {
      // 新增模式或後端未回傳學生資料時，仍需重新載入
      setRefreshKey((prev) => prev + 1)
    }

    return { ok: true }
  } catch (error) {
    return { ok: false, message: error.message || '學生資料儲存失敗' }
  } finally {
    setStudentManageSaving(false)
  }
}
```

## 📊 效能改善對比

### 修改前流程
1. 使用者點擊「更新學生」按鈕
2. 前端呼叫 `updateStudent(payload)` API
3. 後端更新學生資料，回傳 `{success: true}`
4. 前端觸發 `setRefreshKey((prev) => prev + 1)`
5. **重新載入整個學生清單**（包含所有分校）
6. 重新渲染所有相關元件

**問題**：
- API 呼叫次數：2 次（updateStudent + getStudents）
- 資料傳輸量：完整學生清單（可能數十位學生）
- 使用者體驗：需要等待整個清單重新載入

---

### 修改後流程
1. 使用者點擊「更新學生」按鈕
2. 前端呼叫 `updateStudent(payload)` API
3. 後端更新學生資料，回傳 `{success: true, student: {...}}`
4. **前端直接更新該學生的資料**（`setStudents` 中只更新該學生）
5. 只重新渲染受影響的元件

**改善**：
- API 呼叫次數：1 次（只有 updateStudent）
- 資料傳輸量：單一學生資料
- 使用者體驗：即時更新，無需等待

```javascript

```

**適用場景**：
- ✅ 編輯模式 + 後端正常回傳 → 使用方案 A（直接更新）
- ⚠️ 新增模式 → 使用原有邏輯（重新載入，確保完整性）
- ⚠️ 後端未回傳學生資料 → 使用原有邏輯（重新載入，容錯處理）
